# VARIA-Train

> **V**ariational **A**nnotation & **R**endering for **I**nspection **A**I

**VARIA-Train** is the model training module of the broader **VARIA** ecosystem (which includes [VARIA-Swarm](https://github.com/Zidrewndacht/PCBA_AOI_synthdata_export) for synthetic data generation and [VARIA-Iris](https://github.com/Zidrewndacht/PCBA_AOI_AR_Inference) for Intel Edge inference). Originally intended for pre-sliced, synthetically generated PCBA datasets, but this should work for training RF-DETR on any dataset, from a multi-GPU Windows machine with WSL2 Docker. It handles distributed data parallel (DDP) training, dynamic gradient accumulation, and a workaround for WSL2 file I/O bottlenecks.

## Overview

The pipeline is designed to train an object detection model on image slices without relying on naive in-memory augmentations. Because the dataset already contains physics-based environmental and defect variations generated during the *VARIA-Swarm* rendering phase, standard training augmentations are explicitly disabled (this will be updated with *some*, not all, Albumentantions augmentations once they're verified to be useful).

Because training on large image datasets directly from a Windows host via WSL2 mounts is severely bottlenecked by filesystem translation overhead, the pipeline utilizes a **two-stage Docker Compose architecture**. It first synchronizes the dataset into a native Docker volume, then executes the training loop inside an isolated CUDA environment. This ensures fast, native Linux filesystem I/O with NCCL support for DDP00 multi-GPU training.

## Pipeline Components

*   **Data Synchronization (`data-sync` service)**: An Alpine Linux container that checks for a `.new_data_flag` file in the mounted host directory (automatically generated by *VARIA-Swarm* Step 5). If the flag is present, it wipes the destination Docker volume, copies the sliced dataset into it, and burns (deletes) the flag. This prevents redundant multi-gigabyte transfers on every container startup while guaranteeing the training container reads from a high-speed native volume.
*   **Training Execution (`rfdetr_train.py`)**: The core training script utilizing the `rfdetr` library. It dynamically calculates gradient accumulation steps to achieve a target effective batch size across multiple GPUs. It supports automatic resumption from `last.ckpt` if a training run is interrupted, Exponential Moving Average (EMA) weight tracking, and early stopping based on validation metrics.
*   **Container Environment (`Dockerfile`)**: Built on `nvidia/cuda:13.2.1-cudnn-devel-ubuntu24.04` with Python 3.12. It installs the `rfdetr` package with training and logger dependencies, providing a hardware-accelerated environment.

## Configuration

Pipeline behaviors are controlled via `config.yaml` and `docker-compose.yml`.

### Training Parameters (`config.yaml`)
*   **Batch Size Math**: Defines the per-GPU `batch_size`, number of `devices` (GPUs), and a `target_effective_batch_size`. The script automatically calculates the required `grad_accum_steps` to meet the target effective batch size.
*   **Optimization & Stability**: Learning rate, random seed, and EMA toggles.
*   **Early Stopping**: Configurable patience, minimum delta for improvement, and whether to monitor the EMA model weights for stopping criteria.
*   **Checkpointing**: Interval for saving full training states (optimizer, epoch) and a grace period (`skip_best_epochs`) before early stopping logic engages.
*   **Augmentations**: Naive augmentations are strictly disabled (`aug_config={}`), deferring entirely to the pre-rendered synthetic variations from *Varia-Swarm*. This will be changed later to ensure per-epoch dataset variation via limited Albumentations (only those expected to make sense for the task).

### Infrastructure (`docker-compose.yml`)
*   **Host Paths**: The host directory containing the sliced dataset (output from Step 5 of the synthetic data pipeline) must be mapped to the `/src` volume in the `data-sync` service.
*   **Hardware Allocation**: The `rfdetr` service requests all available NVIDIA GPUs. NCCL environment variables are tuned for standard multi-GPU PCIe topologies (disabling P2P and InfiniBand to prevent initialization hangs on WSL2 and consumer/workstation hardware).

## Prerequisites

1.  **Docker & Docker Compose**: Must be installed and running.
2.  **NVIDIA Drivers & Container Toolkit**: Required for GPU passthrough to the training container.
3.  **VARIA-Swarm Output**: A sliced COCO dataset (generated by *Varia-Swarm* Step 5) containing a `.new_data_flag` file to trigger the initial volume sync.

## Usage

1.  **Prepare the Dataset**: Ensure the sliced COCO dataset is placed in the host directory mapped in `docker-compose.yml` (e.g., `D:\!staging\@Mestrado\PCBA-AOI\output_sliced`). Running Step 5 of *Varia-Swarm* automatically creates the `.new_data_flag` that signals the `data-sync` service to update the Docker volume.
2.  **Configure Hyperparameters**: Adjust `config.yaml` to match your hardware (e.g., `devices`, `batch_size`) and training requirements.
3.  **Build and Run**:
    ```batchfile
    docker compose build
    docker compose up
    ```
4.  **Resuming**: If the training is interrupted, simply running `docker compose up` again will automatically detect `last.ckpt` in the output directory and resume from the last saved epoch. Subsequent runs without the `.new_data_flag` will skip the data sync phase and immediately begin training.
